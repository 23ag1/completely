# Dynamic Workflows spike — feasibility for `completely`'s loop engine

**Task:** `claude-harness-p4f` · **Date:** 2026-06-06 · **Method:** docs verification + architectural diff against `run.sh`. The earlier hypothesis (from Opus 4.8 prior to this session) was that DW "natively does what completely hand-rolls — state in Beads not context, parallel disjoint streams, resume." This spike checks the claim against the actual docs and the actual `run.sh` design.

## TL;DR — decision

**Do NOT route `run.sh`'s loop to Dynamic Workflows.** Keep `run.sh`.

Two reasons, in order of weight:

1. **Architectural mismatch at the spine layer.** DW is *one Claude Code session orchestrating ≤16 subagents in script-var state*. `run.sh` is *one bash process orchestrating N independent `claude -p` worker processes, each with fresh context*. Those are different topologies, not "the same thing native vs. hand-rolled":
   - DW's resume is in-session only — "If you exit Claude Code while a workflow is running, the next session starts the workflow fresh" (docs). `run.sh`'s parent is a bash process whose state is in shell vars and is trivially survivable.
   - DW's orchestrator IS itself a Claude session, so the spine consumes context budget on every dispatch decision. `run.sh`'s spine consumes ~zero context — selection runs in Python in the parent.
   - completely's worker contract (fresh-context-per-task, Ralph-style loop shape, the design we already committed to in `flow.md`) is the property DW gives up to deliver in-session resume. They're at *different layers of the system*, not competing implementations of the same layer.

2. **Not invocable in this environment today.** DW is a research-preview feature requiring **Claude Code v2.1.154+** and (on Pro) a `/config` Dynamic-workflows toggle. We are on **v2.1.144**. A live spike here is gated on an upgrade we did not perform in this task (it would replace the running binary mid-session — out of scope for a feasibility check; not a blocker for the decision, which holds independent of version).

The valuable DW slot is **inside a worker**, not as the worker dispatcher — see "Where DW *would* fit" below.

## Architectural diff — DW vs. `run.sh` at the spine layer

Side-by-side against the four properties on which completely picks its loop engine. Sources for `run.sh`: `plugin/scripts/run.sh` (greedy disjoint-zone dispatcher, PID maps in shell vars, fresh `claude -p` per task, commit-before-close gate, evaluator default-FAIL).

| Property | `run.sh` (today) | Dynamic Workflows | Verdict |
|---|---|---|---|
| **Spine cost (context budget)** | ~zero — selection is Python in the parent bash process; the orchestrator is not an LLM. | The orchestrator IS a Claude Code session; each dispatch decision the JS makes via Claude burns parent-session context. JS-only logic (`args`, vars) is free, but the moment Claude reasons over the dispatch (which tasks fit, which writes overlap), that lives in context. | **`run.sh` wins** — completely's whole point is that the spine is deterministic, not an LLM. |
| **State location** | Bash vars: `PID_TASK`, `PID_ZONE`, `PID_LOG`, `NRUN` + Beads as the spine. | JS script vars in the workflow runtime + Beads (if the JS calls `bd`). Matches the design intent. | **Tie on principle** — both keep state out of model context. DW's variant just costs more (see above). |
| **Parallel disjoint streams** | Greedy dispatcher: pick top-of-`bd ready` whose `metadata.write_zone` is disjoint from every running worker and from already-picked zones in this batch. `CMP_PARALLEL=4` default. Same-zone tasks serialize. Tested in `run.sh --self-test` (9 PASS, including prefix-overlap, undeclared-zone-as-global, running-zone block, slot budget, checkpoint skip). | Up to 16 concurrent subagents; disjoint-write-zone scheduling is up to the JS author. Beads `metadata.write_zone` would have to be re-implemented in JS. | **`run.sh` wins on done-work**: disjoint scheduling is already implemented and tested. Re-implementing it in JS gains nothing. |
| **Cross-iteration resume (the Ralph property)** | A fresh `claude -p` per task is a *feature*, not a defect — each worker starts with a clean context window, reads the bead, applies the engine, commits, closes. The bash spine survives indefinitely. | "Resume works within the same Claude Code session." Exit the session → workflow restarts from scratch. completely's design is *exactly* "fresh context per task" — DW's resume model collides with it. | **`run.sh` wins** — DW resume is the wrong shape for our loop. |
| **Worker-contract enforcement** | Step-bound `<<COMPLETELY_ENFORCED ...>>` blocks injected into the worker stdin at spawn time (`plan-check`, `review`, `verify`). Self-test asserts presence + ordering. | DW subagents inherit the parent session's tooling/prompt. Step-bound injection would have to be re-engineered (no obvious hook in docs). | **`run.sh` wins on done-work** — the contract is already wired and unit-tested. |
| **Commit-before-close gate** | `hooks/guard-close.sh` blocks `bd close` on uncommitted tracked changes; enforced via PreToolUse hook in the Claude session, independent of dispatcher. | Same hook would still apply (per-worker), regardless of dispatcher. | **Tie** — orthogonal to the dispatch layer. |
| **Cost of "first usable run"** | Already shipped (`v0.11`), self-test green here (9/9 PASS, see "Spike trace" below). | Requires: (a) Claude Code upgrade, (b) Pro plan toggle, (c) hand-author a JS dispatcher OR have Claude generate one and accept the unknown JS API surface, (d) reimplement disjoint-zone scheduling, (e) reimplement contract injection, (f) accept that resume model fights the fresh-context-per-task principle. | **`run.sh` wins by an order of magnitude** on switching cost. |

The honest read: **DW is not a hand-rolled-loop replacement for `run.sh`**. DW is a *parallel-subagent fan-out primitive*, and `run.sh` is a *parallel-process fan-out spine*. They solve adjacent but different problems.

## Spike trace — falsifiable evidence

Real-runtime path for the existing dispatcher (the thing DW would purportedly replace):

```
$ bash plugin/scripts/run.sh --self-test
run/self-test: dispatcher unit
  PASS disjoint tasks dispatch together (got: 't-a t-b')
  PASS same-zone serializes (got: 's-a')
  PASS prefix overlap detected, unrelated proceeds (got: 'p-a p-c')
  PASS undeclared zone serializes globally (got: 'g-a')
  PASS running zone blocks new overlap (got: 'r-b')
  PASS slot budget honored (got: 't-a')
  PASS checkpoint task not dispatched (got: 'c-b')
  PASS enforced-policy injection: both step blocks present with rule text
  PASS enforced-policy ordering: dispatch < plan-check < review < verify
run/self-test: OK
```

This is what the DW replacement would have to match before being a credible substitute. A native runtime that ships JS orchestration is not a substitute for nine tested behaviors — it's a different layer.

Live DW spike NOT executed: claude `2.1.144 < 2.1.154` gates the native runtime here. The decision does not depend on the live spike — the architectural diff above settles it independent of whether we upgrade.

## Where DW *would* fit (the real opportunity)

DW's real shape — "inside one task, fan out ≤16 subagents with state in script vars" — is a good fit for tasks **inside** a worker, NOT for the dispatcher above it. Concrete candidates inside `completely`:

- **`cmpl bench` matrix runs** — when a single bench task explores N variants in parallel, DW could orchestrate the variants inside one worker session instead of the current "one variant per worker process" pattern. The script-var state is exactly the per-variant accumulator pattern bench wants.
- **`/completely:plan` decomposition** — when planning decomposes a phase into M sub-tasks, DW could fan out the per-sub-task elaboration in parallel inside one planning session.
- **Multi-file refactors inside one task** — when a worker contract's write_zone is broad and the work decomposes into M parallel edits, DW could fan them out.

None of these require touching `run.sh`. They're worker-level skills that *call* DW. That's the right slot for it.
